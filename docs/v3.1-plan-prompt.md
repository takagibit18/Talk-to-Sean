### R2 — Markdown Preview Rendering for Chat Messages

**Current state:** Assistant messages are rendered as raw text in `<p className="whitespace-pre-wrap">`. A response like `**RAG & Retrieval Evaluation**` displays literally, not as bold text.

**Required behavior:** Assistant messages must be rendered through a lightweight Markdown preview component that supports at minimum:

- **Bold** (`**text**`) and *italic* (`*text*`)
- Line breaks and paragraph separation
- Unordered and ordered lists
- Inline code (`` `code` ``)
- Plain URLs rendered as clickable links

**Implementation constraints:**
- Use `react-markdown` (already in the ecosystem, lightweight) or an equivalent small-footprint renderer.
- Only apply to **assistant** messages (`role === "assistant"`). User messages remain plain text.
- Streaming messages: the markdown renderer must handle incomplete/in-progress Markdown gracefully (no broken rendering while tokens are arriving).
- Do NOT render raw HTML (sanitize/disallow `dangerouslySetInnerHTML` equivalent).
- The markdown renderer must not break the streaming cursor (`|`) blink behavior — cursor should appear after the rendered markdown output.
- Keep the `whitespace-pre-wrap` semantics but applied to the rendered output, not raw source.

**Acceptance criteria:**
- [ ] A message containing `**bold text**` displays as **bold text** visually.
- [ ] A message containing `- item 1\n- item 2` displays as a bulleted list.
- [ ] Inline code `` `code` `` renders in a monospace inline style.
- [ ] During streaming, partially-arrived Markdown (e.g., `**bold tex`) does not break the layout or throw errors.
- [ ] The streaming blink cursor `|` is visible at the end of streaming assistant messages.
- [ ] User messages are NOT markdown-rendered (stay plain text).
- [ ] No XSS vector: HTML tags in model output are escaped, not executed.
- [ ] Light and dark themes both render markdown elements legibly.
